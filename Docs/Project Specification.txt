Frontend
Option A (recommended): React + Vite as a PWA. You already know this stack well; PWA gives offline ward use; component model suits the bedside workflow. Add a UI kit (e.g. shadcn/Tailwind) for fast, accessible components.
Backend / rules engine
Option A (recommended): Python + FastAPI. Your strongest language; great for a clean, testable, declarative rules engine and for any ML. Pydantic models map nicely to clinical rule definitions.
Data
Prototype: SQLite/Postgres + local-first store on the client.
Alerting
In-app + push notifications (PWA) for the prototype; pluggable adapter so it can later route to hospital paging/EPMA.
ML
Lightweight glucose-trend prediction (even a simple slope/Kalman filter before any deep model). 
Hosting / infra
Prototype: any standard cloud or local.

Insulin-on-board–aware feed-stop guard. Most tools react to a low reading after it happens. Modelling active insulin against feed state lets the tool warn before the hypo, the moment the feed stops. This is the headline feature and maps directly to the "IV dextrose for active insulin time" guidance.

Built-in audit/research dashboard. The literature literally asks for a national ETF audit tool. Bake time-in-range, hypo/hyper, and adherence reporting in from the start — that's a publishable angle and a selling point to a trust.

Feed-pattern-aware insulin matching. Treat continuous vs bolus vs overnight feeding as distinct scenarios with distinct logic — including the nasty overnight-feed-then-fasting-day transition that catches teams out.

Smart escalation ladder. Unacknowledged high-risk alerts climb the hierarchy with timeouts, so "alert the team" actually reaches someone, with a full who-saw-what trail.

Predictive (advisory) trend alerts. Use CBG slope to pre-empt the next out-of-range reading. Start with simple maths, not deep learning.

Proactive daily titration suggestions. Mirror how a DISN up-titrates: surface a once-a-day "yesterday ran high, consider +X" prompt — the exact behaviour the audit said was missing.
